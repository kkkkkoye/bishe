## 5. 模块一：简单实时客服 WebSocket 方案

## 5.1 模块目标

实现用户进入客服页后，可与商家或管理员进行实时消息通信，消息即时到达并保存聊天记录。

## 5.2 业务场景

典型使用流程：

1. 用户进入商品详情页或客服页
2. 用户点击“联系客服”
3. 小程序建立 WebSocket 连接
4. 用户发送咨询消息
5. 后端接收消息并保存
6. 若客服在线则实时推送回复
7. 若客服不在线则保留离线消息，后续上线后可查看

## 5.3 当前问题与改造思路

### 当前问题

- 当前通过轮询接口拉取消息
- 实时性较差
- 服务器压力随轮询频率升高
- 聊天会话与消息结构不够规范

### 改造思路

采用 “HTTP + WebSocket” 混合模式：

- HTTP：用于拉历史消息、查会话列表、兜底发送消息
- WebSocket：用于实时消息推送

## 5.4 数据库设计

### 方案建议

建议保留现有 `chat` 表用于兼容，但新增更规范的数据表。

### 表 1：客服会话表 `chat_session`

字段建议：

| 字段名 | 类型 | 说明 |
|---|---|---|
| id | bigint | 主键 |
| user_id | bigint | 用户ID |
| service_id | bigint | 客服/管理员ID |
| session_status | varchar(20) | 会话状态，进行中/已结束 |
| last_message | varchar(255) | 最后一条消息 |
| last_time | datetime | 最后消息时间 |
| create_time | datetime | 创建时间 |

### 表 2：聊天消息表 `chat_message`

字段建议：

| 字段名 | 类型 | 说明 |
|---|---|---|
| id | bigint | 主键 |
| session_id | bigint | 会话ID |
| sender_id | bigint | 发送方ID |
| sender_role | varchar(20) | 发送方角色 |
| receiver_id | bigint | 接收方ID |
| content | varchar(1000) | 消息内容 |
| msg_type | varchar(20) | 消息类型，text/image |
| is_read | int | 是否已读 |
| create_time | datetime | 发送时间 |

## 5.5 SQL 设计示例

```sql
CREATE TABLE chat_session (
  id BIGINT PRIMARY KEY,
  user_id BIGINT NOT NULL,
  service_id BIGINT,
  session_status VARCHAR(20) DEFAULT '进行中',
  last_message VARCHAR(255),
  last_time DATETIME,
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE chat_message (
  id BIGINT PRIMARY KEY,
  session_id BIGINT NOT NULL,
  sender_id BIGINT NOT NULL,
  sender_role VARCHAR(20) NOT NULL,
  receiver_id BIGINT NOT NULL,
  content VARCHAR(1000) NOT NULL,
  msg_type VARCHAR(20) DEFAULT 'text',
  is_read INT DEFAULT 0,
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 后端主要职责

#### 1）建立 WebSocket 服务端

职责：

- 接收客户端连接
- 记录在线用户
- 监听客户端发来的消息
- 将消息转发给目标接收人
- 保存消息到数据库

#### 2）维护在线连接池

可使用：

```java
private static final ConcurrentHashMap<Long, Session> ONLINE_USERS = new ConcurrentHashMap<>();
```

说明：

- key：用户ID

#### 3）保存聊天消息

每次收到消息后：

1. 校验发送人
2. 校验会话是否存在
3. 保存消息到 `chat_message`
4. 更新 `chat_session.last_message`
5. 如接收方在线则实时推送

## 6. 模块二：规则化智能推荐方案

## 6.1 模块目标

基于用户行为和商品属性设计一个可解释、轻量级、易实现的规则推荐模块，为用户推荐可能感兴趣的农产品。

## 6.2 建设思路

本项目不建议上复杂机器学习模型，而采用“规则打分”方式实现智能推荐。

优点：

- 实现难度低
- 容易调试
- 推荐原因可解释
- 适合论文描述

## 6.3 可利用数据

推荐逻辑可基于以下业务数据：

- 用户收藏记录
- 用户订单记录
- 商品分类信息
- 商品价格信息
- 商品热销程度

若后续需要更精准推荐，可新增“浏览记录”表。

## 6.4 推荐规则设计

### 核心规则

#### 规则 1：同类偏好推荐

若用户购买或收藏过某类商品，则优先推荐同类商品。

示例：

- 用户买过苹果
- 系统优先推荐红富士、冰糖心苹果、阿克苏苹果

#### 规则 2：热销推荐

若用户行为数据不足，则推荐销量较高商品。

适用情况：

- 新用户
- 无收藏记录用户
- 无订单记录用户

#### 规则 3：价格相近推荐

系统根据用户历史购买均价，优先推荐价格区间相近的商品。

#### 规则 4：收藏相似推荐

用户收藏过某商品后，推荐相同分类或相同产地商品。

#### 规则 5：新上架补充推荐

对权重接近的商品，可优先展示新上架商品，提升商品曝光。

## 6.5 评分机制设计

可采用如下权重：

| 行为/特征 | 分值 |
|---|---|
| 同类订单 | +8 |
| 同类收藏 | +5 |
| 同类浏览 | +3 |
| 热销商品 | +2 |
| 价格接近 | +2 |
| 新上架 | +1 |

最终分值公式：

```text
推荐总分 = 同类订单分 + 同类收藏分 + 同类浏览分 + 热销分 + 价格分 + 新品分
```

按总分降序排列，取前 6 或前 10 个商品返回。

## 7. 模块三：极简版农产品物流跟踪方案

## 7.1 模块目标

在尽量少改现有系统的前提下，实现“订单状态 + 物流说明”的基础物流功能，满足展示、演示和论文描述需求。

## 7.2 当前问题

当前系统已经有 `orders.status` 和 `orders.logistics` 字段，也已有物流页面。

因此最简单的做法不是新建物流轨迹系统，而是直接复用现有字段：

- `status`：显示订单当前状态
- `logistics`：显示物流说明

## 7.3 业务场景

订单物流流程建议简化为：

1. 用户下单
2. 商家确认订单
3. 商家发货并填写物流说明
4. 用户查看订单状态和物流说明
5. 用户确认收货

## 7.4 数据库设计

### 方案建议

本极简版不新增任何物流表，直接复用 `orders` 表现有字段：

- `status`：订单状态
- `logistics`：物流说明

### 推荐字段用法

#### `status`

建议只保留 4 个值，便于开发和答辩：

- 待付款
- 待发货
- 已发货
- 已签收

#### `logistics`

由商家或管理员手动填写一段文字，例如：

```text
已发货，顺丰快递，单号 SF123456789，预计 2 天内送达。
```

或者：

```text
商品已从烟台仓发出，正在运输中，请保持电话畅通。
```

## 7.5 物流状态规范

建议统一使用以下状态：

- 待付款
- 待发货
- 已发货
- 已签收

这样便于：

- 前端展示
- 后端统计
- 论文表述

## 7.6 后端设计

### 建议修改文件

- `src/main/java/com/controller/OrdersController.java`
- `src/main/java/com/entity/OrdersEntity.java`

### 后端功能职责

#### 1）查询订单物流信息

直接根据订单ID查询订单详情，返回：

- `status`
- `logistics`

#### 2）更新物流说明

商家或管理员在订单发货时，直接修改：

- `status = 已发货`
- `logistics = 手工填写的物流说明`

#### 3）确认收货

用户确认收货时：

- `status = 已签收`

## 7.7 接口设计

本极简版不新增物流专用接口，直接复用订单接口。

### 1）查询订单详情

- 路径：`GET /orders/info/{id}` 或现有订单详情接口
- 作用：读取订单状态与物流说明

### 2）修改订单

- 路径：`POST /orders/update`
- 作用：商家更新发货状态与物流说明，用户确认签收

### 返回示例

```json
{
```

## 11. 风险与应对措施

## 11.1 WebSocket 连接不稳定

风险：

- 小程序端断线
- 服务器重启导致连接失效

应对：

- 增加断线重连
- 保留 HTTP 兜底发消息

## 11.2 推荐结果不明显

风险：

- 测试数据不足
- 新用户没有行为记录

应对：

- 增加热销推荐兜底
- 补充模拟测试数据

## 11.3 物流轨迹数据不足

风险：

- 后台未及时录入
- 页面展示为空

应对：

- 设计默认状态
- 允许手工维护物流节点
